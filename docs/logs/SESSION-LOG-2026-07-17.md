- **P1 (found + fixed) — PDF delivery bypass.** `export-pdf.js` tested the deck URL with a case-SENSITIVE `url.startsWith("file://")`. URL schemes are case-insensitive (RFC 3986) and Chromium normalizes them, so `--url FILE:///<unresolved>/deck.html --deck <resolved>/deck.html` made the unresolved deck look like "not a file URL": the deck/url match was skipped, the RESOLVED deck satisfied the gate, and Playwright then printed the UNRESOLVED deck. Reproduced empirically (a PDF of the unresolved deck was produced). The same shape let any non-file URL (`data:`, `about:`) be vouched for by `--deck`. Fixed narrowly: case-insensitive `FILE_URL_RE`; a file:// URL that cannot be resolved now fails closed instead of being treated as "no file URL"; `--deck` must be paired with the matching file:// URL of that same deck; and the deck/url comparison is case-insensitive on Windows (a case-sensitive compare falsely rejected `e:\` vs `E:\`, which are the same file). Added `test_export_pdf_js_gate_survives_url_scheme_case_and_deck_vouching` (failure paths only — no browser needed). Re-verified: all bypass vectors closed, legitimate file:// exports still succeed.

## 2026-07-17.5 — Decision note: content-capacity signal for component selection (pre-implementation)

**Request:** Selection must account for the amount/structure of content a slide needs, not only semantic intent, shape and export fidelity. Research first, decide, then implement the smallest coherent slice.

**Problem observed:** The acceptance deck's "next steps" slide read as visually empty. Not a rendering loss — a content-plan/component-capacity mismatch: the brief's multi-action "next steps" was compressed into ONE CTA sentence to fit `08-next-steps-cta`, whose real capacity is a single headline. Nothing in the pipeline compared *how much content the slide needs* against *how much the component can hold*.

**What the repo already knows (mapped before deciding):**
- `component_type` / `layout_role`: populated on **13/91**. `set_sizes`: **0/91**. `content_shape` on items: **0/91**. All unusable without a 91-item backfill.
- `content_structure`: populated on **89/91** (`label`, `heading`, `body`, `title`, `list-item` ×27, `repeatable-set-of-3` ×4) — well populated but **not trustworthy for capacity**: `08-next-steps-cta` itself declares `list-item` while holding exactly one wrapped headline. That over-tagging is the very thing that produced the sparse slide.
- `text_contract.slot_count`: already in the registry and the retrieval index — but it measures MARKUP, not capacity. The CTA is `slot_count=8` (3 headline fragments of one sentence + 5 chrome); the do-dont checklist is 39. A request for 3 items would "fit" the CTA's 3 headline slots.
- `item_count` already exists on the request schema, but is only compared against `_set_sizes(item)` — empty for every full-slide template, so **no capacity check runs at all** for exactly the items that caused this.

**Decision — capacity signal = `content_blocks`, derived from the component's own text-slot contract:**
1. drop chrome (height <= 0.030 sitting in the top/bottom margin bands: logo, page number, footer, section label — furniture every slide repeats);
2. union-merge remaining slots whose rects fall within a small tolerance (0.012 normalized), so a headline wrapped across several slots collapses to ONE block while list rows separated by real gaps stay distinct;
3. capacity = number of merged blocks.

**Why this signal:** it is grounded in the artifact scaffold actually fills and fidelity actually measures, so it is the operational truth of "what content can this component hold". It needs **no new taxonomy and no backfill**. Validated across all 91 published items before writing any product code: **1** for covers/closings/dividers and `08-next-steps-cta`; 2–3 for section intros/summary grids; **13** `goal-setting-2026.05-process`; **17** `interview-workshop-sunriser.10-do-dont`; **30/33** the checklist tables; **0** for the text-free graphic set. That ordering matches a human reading of the components, and it separates the exact case that failed.

**Rejected alternatives:** `slot_count` (measures markup, not capacity — see the 8-vs-39 trap above); declared `content_structure`/`component_type`/`layout_role` (over-tagged or unpopulated; trusting them means backfilling 91 items); a body-vs-display font-height threshold (arbitrary, font-size dependent); new per-item `capacity` metadata (a broad taxonomy + 91-item backfill, explicitly out of scope).

**Home:** `component-retrieval-index.jsonl`, beside `slot_count` — the index is already documented as carrying "compact buildability facts". One derived artifact regenerates; the registry's 91 items stay untouched.

**Request side:** optional `content_plan` (the structured expansion of the brief: the actual planned items, so preservation is auditable and a "3 next steps" plan cannot be silently reduced to one CTA line). `planned_item_count` = explicit `item_count`, else `len(content_plan)`. When either side is unknown the check is a **no-op** — the same conservative, backward-compatible rule `shape_eligible` already uses.

**Sufficiency now:** it answers exactly the question the gate asks ("can this component hold N distinct content items?") for every published item, deterministically, from data already in the repo. Upper-bound capacity (tiny content in a huge layout) is deliberately NOT gated in this slice: a cover/divider is intentionally sparse, so an upper bound would reject correct designs. Capacity is recorded on every candidate so that case stays visible — noted as residual risk.

**State:** Decision note only — no code changed at this point.
